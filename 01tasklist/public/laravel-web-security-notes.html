01

CSRF 是什么 & Laravel 如何防护

攻击原理
跨站请求伪造（CSRF）

攻击者诱导已登录用户访问恶意页面，页面自动向目标网站发送伪造请求。浏览器会自动带上 Cookie，服务器无法区分真假请求。

攻击成立的前提

用户已登录目标网站 + 目标网站没有 CSRF 防护 + 用户被诱导访问了恶意页面

Laravel 防护方案
在 Blade 表单中添加 @csrf

自动生成 <input type="hidden" name="_token" value="...">

AJAX 请求携带 Token

通过 meta[name=csrf-token] + 请求头 X-CSRF-TOKEN，或利用 Axios 自动读取 XSRF-TOKEN Cookie

排除特定路由（Webhook 等）

在 VerifyCsrfToken 中间件的 $except 数组中声明

API 路由无需 CSRF

routes/api.php 使用 api 中间件组，不含 CSRF 检查，改用 Token 认证

⚠

验证失败返回 419 — 常见原因：缺少 @csrf、Session 过期、AJAX 未携带 Token、路由文件写错位置

02

同源策略 — 为什么恶意网站拿不到 Token

同源的定义

协议 + 域名 + 端口 三者完全相同才算同源。evil.com 与 bank.com 不同源，浏览器严格隔离。

恶意网站能做到

让用户浏览器向 bank.com 发送 POST 请求

浏览器自动附带 bank.com 的 Cookie（含登录凭证）

恶意网站做不到

读取 bank.com 页面的 DOM（Token 藏在隐藏字段里）

读取 bank.com 的 Cookie（按域隔离）

💡

关键不对称性：Cookie 是浏览器自动携带的（你没得选），但读取另一个域的 Cookie 或 DOM 被同源策略严格禁止。CSRF Token 正是利用了这个不对称性。

核心模型：Token 是随机钥匙，服务端存原件，客户端提交副本，比对是否一致。没有推导算法，无从破解。

为什么极难破解 — 四层保障
密码学随机性

random_bytes(40) = 320 位随机数，每秒猜 10 亿次也需要宇宙年龄才能碰对

Token 绑定 Session，每人不同

即使猜中一个用户的 Token，对其他用户无效，Session 过期后立即失效

hash_equals() 防时序攻击

强制走完整比较，无论第几个字符不同耗时完全一样，无法通过响应时间"逐字试探"

Token 轮换

每次成功响应后重新生成，旧 Token 立即失效，截获旧 Token 也无用

CSRF 防护失效的场景
漏洞场景原因状态
网站存在 XSS 漏洞攻击者注入 JS 直接读取 DOM 里的 Token高危
使用 HTTP 而非 HTTPS中间人可嗅探 Token 明文高危
Token 泄露在 URLReferer 头、日志、缓存都会记录中危
$except 配置不当开发者手动排除了不该排除的路由中危
04

Cookie 与 Session 的本质区别

💡

一句话：Cookie 是信使，Session 是保险箱。Cookie 只负责把 Session ID 带给服务器，真正的数据锁在服务器的 Session 里。

维度CookieSession
数据存在哪浏览器本地服务器（文件/Redis/DB）
容量4 KB无限制（服务器决定）
安全性客户端可见，需加密客户端只见 ID，数据不暴露
自动携带每次请求自动带ID 通过 Cookie 带，数据在服务端
关浏览器后无 Expires 则消失ID Cookie 消失，服务端数据可能还在
Cookie 重要属性
属性作用
HttpOnlyJS 无法读取，防 XSS 盗取 Cookie
Secure只在 HTTPS 连接下发送
SameSite=Lax阻止跨站请求携带，辅助防 CSRF
Expires/Max-Age持久化 Cookie；不设则关闭浏览器失效
Session 与 Cookie 不是缺一不可
Cookie 传 Session ID（最常用）

Web 应用标准方案，HttpOnly + Secure 保护，Laravel 默认方式

URL 参数传 ID（不推荐）

会出现在日志、历史、Referer 头中，安全性差，现代应用不应使用

请求头传 JWT（API/SPA 常用）

Authorization: Bearer token，服务端无状态，Laravel Sanctum Token 模式

Laravel Session 配置要点
// config/session.php
'driver'          => 'redis',      // 存储后端
'lifetime'        => 120,         // 服务端数据保留分钟数
'expire_on_close' => false,       // true=关浏览器即过期
'http_only'       => true,        // JS 读不到（默认已开启）
'secure'          => true,        // 生产环境必须开启
'same_site'       => 'lax',       // 跨站限制
⚠

存储在哪（driver）和何时过期（lifetime / expire_on_close）是两个独立配置，互不影响。

05

Session ID 会被窃取吗

攻击方式手段防御
XSS 注入脚本 document.cookie 读取后外传 HttpOnly Cookie
网络嗅探/中间人 HTTP 明文截获，公共 WiFi 高危 Secure Cookie + HTTPS
共享设备/物理访问 直接查看或复制 Cookie 文件 Session 过期 + 登出销毁
子域名 Cookie 泄露 evil.yourapp.com 读取父域 Cookie 精确 Domain + SameSite=Strict
会话固定攻击 预先植入 Session ID，等用户登录 登录后 regenerate()
Laravel 登录后的 Session 处理
// 登录成功后换 ID（防会话固定攻击）
$request->session()->regenerate();

// 退出登录时完全销毁
Auth::logout();
$request->session()->invalidate();
$request->session()->regenerateToken();
✓

HttpOnly + Secure + HTTPS 三件套 + 合理过期策略，已覆盖绝大多数真实攻击场景。剩余风险本质是 XSS 和物理安全问题，不是 Session 机制本身的缺陷。

07

Session 持久化 & 数据库多行问题

为什么要用 Redis/数据库存 Session
多台服务器的问题

文件/内存存储时，用户第1次打到服务器A，第2次打到服务器B，B上没有Session → 被踢出登录

Redis 共享存储解决问题

所有服务器读写同一个 Redis，Session 数据共享，用户无感知

Redis TTL 自动清理

按 lifetime 自动过期删除，无需额外任务。数据库方案需定期运行 php artisan session:gc

数据库出现 2 行的原因
原因说明判断方法
最常见 登录触发 regenerate() 访客 Session 行残留 + 登录后新行创建 旧行 user_id = NULL
多浏览器同时登录 Chrome 一行，Firefox 一行，都有效 两行 user_id 都有值
过期数据未 GC GC 概率触发（默认 2/100），旧行未及时清理 旧行 last_activity 很久之前 
-- 快速排查
SELECT id, user_id, FROM_UNIXTIME(last_activity) as time
FROM sessions
ORDER BY last_activity DESC;
💡

user_id = NULL 的行是访客遗留，等 GC 自动清理。手动清理运行 php artisan session:gc

存储位置 vs 过期时间 — 两个独立维度
expire_on_closelifetime效果
true120 分钟关浏览器立刻登出，最多撑 120 分钟
false120 分钟不活跃 120 分钟后登出，关浏览器没影响
false43200 分钟"记住我"效果，30 天内免登录
